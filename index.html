Yeni Başlayanlar İçin Proje Yapısı & Repository Rehberi

Açık kaynak, çok kullanıcılı projelerde düzenli repo yapısı, katkı süreci ve Gitflow kullanımını hızlıca anlatır.

1) GitHub Org & Repo 2) .gitignore 3) README.md 4) CONTRIBUTING.md 5) Issue/PR Templates 6) Branch Strategy (Gitflow) 7) GitHub Projects Hızlı Checklist

Hedef: Yeni katılan bir kişi bu dokümanı okuyarak projeyi klonlayıp çalıştırabilsin, branch/PR sürecini doğru uygulasın ve iş takibini (Projects board) anlayabilsin.

Not: Komutlar örnektir. Projene göre isimleri/versiyonları uyarlayabilirsin.

1) GitHub Organization & Repositories

GitHub Organization

  • Projelerin tek çatı altında toplanması (repo’lar, takımlar, izinler).
  • Team bazlı yetkilendirme önerilir (Admin/Maintainer/Contributor vb.).
  • Açık kaynakta; Issues/Discussions/Wiki gibi modüller netleştirilmelidir.

Repository (Repo) İsimlendirme

  • Net ve kısa: customer-api, admin-panel, mobile-app
  • Monorepo ise: tek repo içinde paketler/servisler klasörlenir.

Önerilen repo içeriği

  • README.md — kurulum ve kullanım
  • CONTRIBUTING.md — katkı süreci
  • LICENSE — açık kaynak lisansı
  • CODE_OF_CONDUCT.md — topluluk kuralları
  • SECURITY.md — güvenlik bildirim süreci
  • .github/ — templates, workflows, config

Minimum GitHub ayarları

  • Default branch: develop (veya ekip kararına göre main)
  • Branch protection: main (ve çoğu zaman develop)
  • PR zorunluluğu, en az 1 review, status check’ler
  • Issue/PR şablonları
Komutlar: Repo klonlama & ilk kurulum 
# Repo klonla
git clone https://github.com/ORG/REPO.git
cd REPO

# Remote kontrol
git remote -v

# Branch listesi
git branch -a

2) .gitignore Dosyaları

.gitignore, repoya girmemesi gereken dosyaları listeler. Örn: build çıktıları, IDE ayarları, local config, cache.

Öneri: Teknoloji + OS + IDE’yi kapsayan bir .gitignore kullanın.

Hazır şablonlar: github/gitignore deposu.

Tipik örnekler

  • .NET: bin/, obj/, *.user
  • Node/React: node_modules/, dist/, .next/
  • Mac: .DS_Store — Windows: Thumbs.db
  • IDE: .idea/, .vscode/ (projeye göre)
Komutlar: ignore edilen dosyaları kontrol 
# .gitignore sonrası takipte kalan dosyaları temizlemek gerekiyorsa:
git rm -r --cached .
git add .
git commit -m "chore: apply gitignore"

Dikkat: Bu komut repodaki dosyaları silmez, sadece git indeksinden çıkarıp kuralları yeniden uygular.

3) README.md (Kurulum & Kullanım Rehberi)

README, repoya giren herkesin ilk okuyacağı dosyadır. Yeni başlayan biri “Bu proje nedir ve nasıl çalıştırırım?” sorusuna burada cevap bulmalı.

README içeriği (öneri)

  • Proje özeti + amaç
  • Teknolojiler & gereksinimler (versiyonlar)
  • Kurulum adımları
  • Çalıştırma (dev/prod)
  • Environment variables (.env.example)
  • Test komutları + lint/format
  • Deployment / Docker (varsa)
  • Linkler: Swagger, dokümanlar, board
Komutlar: Örnek kurulum/çalıştırma blokları 
# Node / React
npm ci
npm run dev

# .NET
dotnet restore
dotnet build
dotnet run --project src/Your.Api

# Test
npm test
dotnet test

# Docker (varsa)
docker compose up -d --build

İpucu: README’deki komutlar “kopyala-yapıştır” ile çalışmalı. Eksik adım bırakmayın.

4) CONTRIBUTING.md

CONTRIBUTING, projeye nasıl katkı yapılacağını netleştirir. Çok kullanıcılı (multi-contributor) açık kaynakta standarttır.

CONTRIBUTING içinde olması önerilenler

  • Branch açma kuralları ve isimlendirme
  • Commit mesaj formatı
  • PR süreci: açıklama, test, checklist
  • Code review beklentisi (kim, kaç onay, SLA)
  • Lint/format kuralları
  • Issue açma kuralları
Komutlar: Fork tabanlı katkı akışı (açık kaynak için) 
# 1) Fork'ladığın repoyu klonla (kendi hesabın)
git clone https://github.com/SEN/REPO.git
cd REPO

# 2) Upstream ekle (asıl repo)
git remote add upstream https://github.com/ORG/REPO.git
git fetch upstream

# 3) Develop'i güncelle
git checkout develop
git pull upstream develop

# 4) Feature branch aç
git checkout -b feature/short-title

# 5) Değişiklikleri commit et, pushla
git add .
git commit -m "feat: add login endpoint"
git push -u origin feature/short-title

5) Issue / Pull Request Template’leri

Template’ler, eksik bilgiyle açılan issue/PR’ları azaltır. Herkes aynı formatta yazar, maintain etmek kolaylaşır.

Issue Template (öneri başlıkları)

  • Problem açıklaması
  • Beklenen davranış
  • Mevcut davranış
  • Tekrarlama adımları
  • Log / ekran görüntüsü
  • Ortam bilgisi (OS, sürüm)

PR Template (öneri başlıkları)

  • Ne değişti?
  • Neden gerekli?
  • Nasıl test edildi?
  • Ekran görüntüsü / çıktı (UI ise)
  • Checklist: test/lint/docs
Dosya konumları (GitHub standardı) 
.github/
  ISSUE_TEMPLATE/
    bug_report.yml
    feature_request.yml
  pull_request_template.md

6) Branch Strategy (Gitflow)

Gitflow, ekipte paralel geliştirmeyi düzenler. Temel amaç: ana branch’leri korumak ve değişiklikleri kontrollü taşımak.

Temel branch’ler

  • main → Production (canlı)
  • develop → Günlük geliştirme
  • feature/* → Yeni özellik
  • hotfix/* → Canlıdaki acil düzeltme
  • release/* (opsiyonel) → sürüm hazırlığı

Kurallar (öneri)

  • main’e doğrudan commit yok → sadece PR
  • PR için CI (test/lint) zorunlu
  • En az 1 review + resolve comments
  • Merge yöntemi: Squash (sıklıkla önerilir) veya ekip standardı
Komutlar: Feature akışı (develop üzerinden) 
# Develop'i güncelle
git checkout develop
git pull origin develop

# Feature branch
git checkout -b feature/user-login

# Çalış, commit et
git add .
git commit -m "feat: add user login flow"

# Push
git push -u origin feature/user-login

# PR aç: feature/user-login -> develop
Komutlar: Hotfix akışı (main üzerinden acil düzeltme) 
# Main'i güncelle
git checkout main
git pull origin main

# Hotfix branch
git checkout -b hotfix/critical-bug

# Fix, commit, push
git add .
git commit -m "fix: prevent null reference on startup"
git push -u origin hotfix/critical-bug

# PR aç: hotfix -> main (sonra main'den develop'a geri merge/cherry-pick)

7) GitHub Projects Board

İş takibini görsel ve ortak bir yerde tutar. Yeni başlayan biri “şu an ne üzerinde çalışılıyor?” sorusuna board’dan bakabilmeli.

Önerilen kolonlar

  • Backlog → yapılacaklar
  • In Progress → üzerinde çalışılanlar
  • Review → code review bekleyenler
  • Done → tamamlananlar

Öneri: Issue’ları board’a bağlayın ve PR’ları Issue’lara linkleyin. Böylece ilerleme izlenebilir olur.

Kısa not: PR açıklamasında “Fixes #123” gibi ifadeler issue’yu otomatik kapatmaya yardımcı olabilir (repo ayarlarına bağlı).

Hızlı Checklist (Yeni Katılan İçin)

  1. Repo’yu klonla: git clone
  2. Gereksinimleri kur: Node/.NET/Docker vb.
  3. README adımlarını uygula: npm ci / dotnet restore
  4. Projeyi çalıştır: npm run dev / dotnet run
  5. Board’a bak: aktif iş var mı?
  6. Yeni iş için branch aç: feature/...
  7. PR aç, template’e göre doldur, review iste
Komutlar: Günlük rutin (sync + temizlik) 
# Upstream/develop güncelle
git checkout develop
git pull origin develop

# Lokal branch’leri kontrol et
git branch

# Uzak branch’leri güncelle
git fetch --all --prune

# Rebase yerine merge politikası ekibinize göre değişebilir
